# Lernziele – Funktionale Programmierung (FP)

## 1) Grundlagen: Paradigma der Funktionalen Programmierung

**Funktionale Programmierung (FP)** ist ein Programmierparadigma, bei dem Berechnungen als Auswertung von Funktionen modelliert werden.

Kernidee:
- Statt den Zustand von Variablen ständig zu ändern, werden Werte über Funktionen in neue Werte transformiert.
- Fokus auf **Was** berechnet werden soll (Deklaration), weniger auf **Wie** jeder einzelne Schritt den Zustand verändert.

Wichtige FP-Prinzipien:
- Pure Functions
- Unveränderlichkeit (Immutable Values)
- Referenzielle Transparenz
- Vermeidung von Seiteneffekten
- Komposition kleiner Funktionen

---

## 2) Vor- und Nachteile der FP

### Vorteile
1. **Weniger Bugs**
   - Keine versteckten Zustandsänderungen -> Verhalten ist besser vorhersagbar.
2. **Einfacher zu testen**
   - Pure Functions haben für gleiche Inputs immer gleichen Output.
3. **Bessere Parallelisierung**
   - Ohne gemeinsamen veränderbaren Zustand sind Race Conditions seltener.
4. **Wiederverwendbarkeit**
   - Kleine, klar definierte Funktionen lassen sich gut kombinieren.

### Nachteile
1. **Einstiegshürde**
   - Denkweise unterscheidet sich stark von imperativ/prozedural.
2. **Performance-/Speicherkosten (je nach Sprache/Umsetzung)**
   - Viele neue Objekte statt In-Place-Änderungen.
3. **I/O bleibt notwendig**
   - Reale Programme brauchen Seiteneffekte (Datei, DB, Netzwerk), daher oft Mischstil.
4. **Rekursion kann schwierig sein**
   - Ohne Tail-Call-Optimierung droht Stack-Overflow.

---

## 3) Prozedurale / Imperative / Deklarative Programmierung

### Prozedural

Ist ein Teil des **imperativen Paradigmas**. Man kenn es ich wie ein Konzeptrezept vorstellen, das der Computer Punkt für Punkt abarbeitet.

- Programm als Folge von Prozeduren/Anweisungen.
- Fokus auf Ablaufsteuerung.
- Es wird intensiv mit veränderbaren Zuständen (Variablen), Schleifen und Zuweisungen gearbeitet.
- Das "**WIE**" steht hier, im Gegensatz zum "**WAS**" im **deklarativen** Paradigma, im Vordergrund.

### Imperativ
- Beschreibt **wie** der Computer etwas Schritt für Schritt tun soll.
- Nutzt veränderbaren Zustand, Schleifen, Zuweisungen.

### Deklarativ
- Beschreibt **was** erreicht werden soll.
- Weniger explizite Steuerung des Ablaufs.
- FP ist überwiegend deklarativ.

### Kurzvergleich
- Imperativ: „Nimm Liste, lauf durch, erhöhe Zähler, ändere Array.“
- Deklarativ/FP: „Mappe Liste auf neue Werte.“

## 4) Pure vs. Impure Functions

### Pure Function
Eine Funktion ist **pure**, wenn:
1. gleicher Input -> immer gleicher Output
2. keine Seiteneffekte außerhalb der Funktion

Beispiel (pure):

```js
const add = (a, b) => a + b;
```

### Impure Function
Eine Funktion ist **impure**, wenn sie z. B.:
- globale Variable ändert,
- auf aktuelle Zeit/Zufall zugreift,
- I/O macht (Konsole, Datei, DB, Netzwerk),
- Input-Objekt mutiert.

Beispiel (impure):

```js
let counter = 0;
const inc = () => ++counter; // verändert externen Zustand
```

---

## 5) Regeln von Pure Functions (Prüf-Checkliste)

**Eine** Funktion ist nur dann pure, wenn alle Punkte erfüllt sind:
- [ ] Nur Parameter als Input
- [ ] Kein Zugriff auf veränderbaren externen Zustand
- [ ] Keine Mutation von Parametern
- [ ] Kein I/O in der Funktion
- [ ] Deterministisches Ergebnis

Merksatz:
**Pure = deterministisch + nebenwirkungsfrei**

---

## 6) Wann ist es pure, wann impure?

## Pure

```js
const square = x => x * x;
```

- Nur Input `x`, immer gleiches Resultat, keine Nebeneffekte.

## Impure wegen Zufall

```js
const randomId = () => Math.random();
```

- Gleicher Aufruf liefert unterschiedliche Ergebnisse.

## Impure wegen Zeit

```js
const now = () => Date.now();
```

## Impure wegen Mutation

```js
const pushItem = (arr, x) => { arr.push(x); return arr; };
```

## Pure Alternative zur Mutation

```js
const addItem = (arr, x) => [...arr, x];
```

## 7) Referenzielle Transparenz

Ein Ausdruck ist **referenziell transparent**, wenn man ihn jederzeit durch seinen Wert ersetzen kann, ohne das Programmverhalten zu ändern.

Beispiel:

```js
const double = x => x * 2;
const y = double(4) + 1; // 8 + 1
```

`double(4)` kann durch `8` ersetzt werden -> Verhalten bleibt gleich.

Bei impuren Funktionen geht das nicht zuverlässig.

---

## 8) Seiteneffekte

Ein **Seiteneffekt** ist jede beobachtbare Änderung außerhalb der Funktionsgrenzen.

Beispiele:
- Console-Ausgabe
- Datei schreiben
- Datenbank-Update
- HTTP-Request
- Änderung globaler Variablen

FP-Strategie:
- Kernlogik möglichst pure halten.
- Seiteneffekte an den „Rändern“ bündeln (z. B. in Controller/Service/I/O-Schicht).

---

## 9) Immutable Values – Konzept

**Immutable** bedeutet: Nach Erstellung wird ein Wert nicht verändert, sondern bei Änderungen als neuer Wert erzeugt.

Vorteile:
- Kein versehentliches Überschreiben
- Besseres Debugging
- Thread-sicherer

Beispiel:

```js
const original = { name: "Ana", points: 10 };
const updated  = { ...original, points: 11 };
```

`original` bleibt unverändert, `updated` ist neues Objekt.

---

## 10) Call by Reference / Call by Value

Wichtig: Viele moderne Sprachen (z. B. JavaScript) sind technisch **Call by Value**, aber bei Objekten wird der Referenzwert kopiert.

### Call by Value
- Eine Kopie des Werts wird übergeben.
- Änderungen an der lokalen Variable ändern den Aufrufer nicht direkt.

### „Referenz-Effekt“ bei Objekten
- Der Referenzwert wird kopiert.
- Beide zeigen auf dasselbe Objekt.
- Mutation innerhalb der Funktion wirkt nach außen.

Beispiel:

```js
function mutate(obj) {
  obj.x = 99; // verändert dasselbe Objekt
}

const a = { x: 1 };
mutate(a);
// a.x ist jetzt 99
```

FP-Regel:
- Keine Mutation von übergebenen Objekten.
- Stattdessen neues Objekt zurückgeben.

---

## 11) Rekursion – Konzept verstehen

**Rekursion**: Eine Funktion ruft sich selbst auf, bis ein Abbruchfall (Base Case) erreicht wird.

Struktur:
1. Base Case (Abbruch)
2. Rekursiver Fall (Selbstaufruf mit kleinerem Teilproblem)

Beispiel Fakultät:

```js
function fac(n) {
  if (n <= 1) return 1;      // Base Case
  return n * fac(n - 1);     // Rekursiver Fall
}
```

`fac(4)` -> `4 * 3 * 2 * 1 = 24`

---

## 12) Eine Rekursion programmiert (Beispiel)

Summe einer Liste rekursiv:

```js
function sum(arr) {
  if (arr.length === 0) return 0;
  return arr[0] + sum(arr.slice(1));
}
```

Ergebnis:
- `sum([1,2,3,4])` -> `10`

Hinweis:
- Bei sehr großen Listen oft iterative oder optimierte Varianten nutzen.

---

## 13) Code erkennen und richtig interpretieren – Kurzleitfaden

Wenn du FP-Code liest, prüfe in dieser Reihenfolge:
1. **Input/Output klar?**
2. **Pure oder impure?**
3. **Mutiert etwas bestehende Daten?**
4. **Seiteneffekte vorhanden?**
5. **Referenziell transparent?**
6. **Falls rekursiv: Base Case korrekt?**

---

## Mini-Zusammenfassung (Prüfungsreif)

- FP denkt in Funktionen und Daten-Transformationen.
- Pure Functions + Immutable Values sind Kernbausteine.
- Referenzielle Transparenz macht Code vorhersagbar.
- Seiteneffekte sind nötig, sollten aber isoliert werden.
- Rekursion ist ein zentrales Werkzeug in FP.
- In realen Projekten ist oft ein Mischstil (FP + imperativ) sinnvoll.
